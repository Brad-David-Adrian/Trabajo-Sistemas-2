Mini-shell UFV - Guía de Defensa

Objetivo del Proyecto
Implementar un shell interactivo básico que demuestre conceptos de sistemas operativos: gestión de procesos, tokenización de entrada, comandos built-in vs. externos, y resolución de PATH.

Estructura General del Código
- **main()**: Bucle REPL (Read-Eval-Print Loop) con prompt "ufv: ", lee entrada con fgets, tokeniza, ejecuta comando y repite.
- **Tokenización**: Maneja espacios, comillas simples/dobles y escapes (\) para dividir la línea en tokens.
- **Tabla de comandos**: cmd_table[] asocia nombres de built-ins con funciones.

Comandos Built-in

1. **exit**
   - **Propósito**: Termina la shell inmediatamente.
   - **Implementación**: Llama a exit(0) sin argumentos. Debe ser built-in porque no puede ejecutarse en un proceso hijo (el hijo terminaría, pero el padre seguiría).
   - **Ejemplo**:
     ```
     ufv: exit
     ```
     Salida: El shell se cierra.

2. **pwd** (Print Working Directory)
   - **Propósito**: Muestra el directorio de trabajo actual.
   - **Implementación**: Usa getcwd() para obtener el path absoluto. Si falla, muestra error con perror("pwd").
   - **Por qué built-in**: Los cambios de directorio deben afectar al proceso padre, no al hijo.
   - **Ejemplo**:
     ```
     ufv: pwd
     /Users/brad/Desktop/SI2/Trabajo-Sistemas-2
     ufv:
     ```

3. **cd** (Change Directory)
   - **Propósito**: Cambia el directorio de trabajo actual.
   - **Implementación**: Usa chdir(). Si no se da argumento, va a HOME (getenv("HOME")). Maneja errores con perror("cd").
   - **Por qué built-in**: Similar a pwd, para que el cambio persista en la shell padre.
   - **Ejemplos**:
     ```
     ufv: cd ..
     ufv: pwd
     /Users/brad/Desktop/SI2
     ufv: cd /tmp
     ufv: pwd
     /tmp
     ufv: cd
     ufv: pwd
     /Users/brad
     ```

Ejecución de Programas Externos
- **Propósito**: Ejecutar comandos no built-in como programas del sistema.
- **Implementación**:
  - Crea proceso hijo con fork().
  - En el hijo: execvp() busca en PATH; si falla, run_program_thru_path() (manual con strtok_r).
  - Padre espera con waitpid() para evitar zombies.
  - Si no se encuentra: "ufv: <comando>: command not found".
- **Manejo de PATH**: execvp() usa la variable de entorno PATH automáticamente. El fallback manual demuestra comprensión de cómo funciona.
- **Ejemplos**:
  - Con PATH:
    ```
    ufv: ls -la
    total 16
    drwxr-xr-x  5 brad staff  160 Mar 17 10:00 .
    drwxr-xr-x  3 brad staff   96 Mar 17 09:00 ..
    -rw-r--r--  1 brad staff  512 Mar 17 10:00 README.md
    ...
    ```
  - Con path absoluto:
    ```
    ufv: /bin/echo "Hola Mundo"
    Hola Mundo
    ```
  - Comando no encontrado:
    ```
    ufv: nonexistentcommand
    ufv: nonexistentcommand: command not found
    ```

Manejo de Errores y Robustez
- **Validaciones**: tokens == NULL o longitud 0 saltan al prompt sin crash.
- **Memoria**: args liberado en padre e hijo; tokens_destroy() libera todo.
- **Errores**: perror() para fork, malloc, chdir, etc. Códigos de retorno propagados.
- **Input especial**: Líneas vacías, solo espacios, o con comillas/escapes manejadas correctamente.

Aprendizajes Clave
- Flujo de una shell: parsing → built-in check → fork/exec → wait.
- Importancia de gestión de procesos y memoria en C.
- Diferencia entre built-ins (efectos en padre) y externos (en hijo).

Diagrama de Flujo Simplificado
1. Usuario ingresa línea → fgets()
2. tokenize() → array de tokens
3. Si vacío → mostrar prompt
4. lookup() en cmd_table:
   - Match: ejecutar función built-in
   - No match: fork() → execvp() en hijo → waitpid() en padre
5. Mostrar prompt y repetir

Resultado
Este mini-shell demuestra implementación completa de un intérprete de comandos básico, con énfasis en procesos, PATH y robustez, ideal para evaluar conocimientos en SO y C.
